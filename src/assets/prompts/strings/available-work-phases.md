name: "available-work-phases"
---
* Engineering: Use this Work Phase to write code, run unit tests, run functional tests, verify end-to-end functionality, or otherwise evaluate or modify a software project's files and functionalityt. That includes creating, deleting, moving or renaming files or directories, building projects, and linting code to build features or fix bugs. Use this for implementation and functional correctness. If the task requires tuning or theoretical analysis, use Research instead. If no other Work Phase is clearly more suitable, this should be the default choice.
* Planning: To create a plan to solve larger projects, problems, or tasks. Do NOT use this phase to execute a plan, it will only return a plan, typically in the form of creating a Project_Plan.md file. It is designed to take large and complex jobs and break them down into simpler step-by-step tasks to be solved by other Work Phases.
* Documentation: To write documents. This Work Phase will return a well written, well formatted document in markdown.
* Validation: Use this Work Phase ONLY to validate that the project has been successfully completed and fully meetss the initial requirements, returning a report on the validation result. Do NOT use this phase to find, edit or create files, run tests, or verify end-to-end functionality.
* Code Review: To perform a code review that will scrutinize source code for formatting, syntax, style quality, and runnability. Will fix any problems in the code and return a detailed feedback document with findings. Use this Work Phase to do a code review and confirm code is correctly written.
* Research: To perform investigations, mathematical modeling, and automated experimentation. Use this phase to optimize system parameters (tuning), stress-test solutions for robustness (Monte Carlo), or analyze trade-offs between different engineering approaches. This Work Phase returns data-driven findings, optimal configuration values, and evidence-based recommendations.
